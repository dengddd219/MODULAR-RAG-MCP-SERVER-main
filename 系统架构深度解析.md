## 第一部分：模块总体定位

### 1.1 Dense Retriever 在整个系统架构中的角色

**Dense Retriever** 是 RAG pipeline 中的**语义检索核心组件**，位于 `src/core/query_engine/dense_retriever.py`。

#### 架构位置图

```
┌─────────────────────────────────────────────────────────┐
│                    User Query                           │
│              "如何配置 Azure OpenAI？"                    │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│              QueryProcessor                            │
│  - 关键词提取                                            │
│  - 停用词过滤                                            │
│  - Filter 解析                                           │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
         ┌───────────┴───────────┐
         │                       │
         ▼                       ▼
┌─────────────────┐    ┌─────────────────┐
│ DenseRetriever  │    │ SparseRetriever  │
│ (语义检索)       │    │ (关键词检索)      │
└────────┬─────────┘    └────────┬─────────┘
         │                       │
         ▼                       ▼
┌─────────────────────────────────────────┐
│         HybridSearch (RRF Fusion)       │
└────────────────────┬────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────┐
│         CoreReranker (可选)              │
└────────────────────┬────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────┐
│      RetrievalResult[] → LLM            │
└─────────────────────────────────────────┘
```

#### 核心职责

1. **语义理解**：将用户查询转换为高维向量（embedding），捕获语义含义
2. **相似度搜索**：在向量库中查找与查询向量最相似的文档块（chunks）
3. **结果标准化**：将向量库返回的原始结果转换为统一的 `RetrievalResult` 对象

### 1.2 在 RAG Pipeline 中的阶段

Dense Retriever 属于 **Retrieval（检索）阶段**，具体位置：

```
Ingestion Pipeline (离线)
  ↓
  [Document] → [Chunks] → [Embeddings] → [Vector Store]
  
Query Pipeline (在线)
  ↓
  [User Query] → [Query Embedding] → [DenseRetriever] → [Top-K Chunks]
                                                              ↓
                                                      [LLM Generation]
```

**关键点**：
- **离线阶段**：文档被切分、编码、存储到向量库（ChromaDB）
- **在线阶段**：查询被编码，DenseRetriever 执行相似度搜索

### 1.3 解决的核心问题

#### 问题 1：语义匹配 vs 字面匹配

**传统关键词搜索的局限**：
- 查询："如何设置 API 密钥？"
- 文档："配置 API Key 的方法"
- **问题**：关键词不重叠（"设置" ≠ "配置"，"密钥" ≠ "Key"），但语义相同

**Dense Retrieval 的优势**：
- Embedding 模型将"设置"和"配置"映射到向量空间中相近的位置
- 即使字面不匹配，也能通过向量相似度找到相关文档

#### 问题 2：多语言与同义词

- 查询："RAG 系统架构"
- 文档："Retrieval-Augmented Generation 设计"
- Dense 能理解缩写、同义词、多语言表达

#### 问题 3：上下文理解

- 查询："Azure OpenAI 的 embedding 模型"
- 文档："text-embedding-ada-002 是 Azure 提供的向量化模型"
- Dense 能理解"embedding 模型"与"向量化模型"的语义等价性

### 1.4 为什么选择 Dense Retrieval？

#### vs BM25（Sparse Retrieval）

| 维度 | Dense Retrieval | BM25 (Sparse) |
|------|----------------|---------------|
| **匹配方式** | 语义相似度（向量空间） | 词频统计（倒排索引） |
| **优势** | 理解同义词、语义等价 | 精确匹配、低延迟 |
| **劣势** | 需要 Embedding API、计算成本高 | 无法处理语义变化 |
| **适用场景** | 概念查询、多语言、同义词 | 精确术语、代码搜索 |

#### vs Keyword Search

- **Keyword Search**：完全字面匹配，无法处理语义变化
- **Dense Retrieval**：语义理解，但可能召回不精确的文档

#### vs Hybrid Search（当前系统采用）

**为什么系统同时使用 Dense + Sparse？**

1. **互补性**：
   - Dense 擅长语义理解，但可能漏掉精确术语
   - Sparse 擅长精确匹配，但无法处理同义词
   - **Hybrid = 两者结合，取长补短**

2. **RRF Fusion 算法**：
   - 将两个检索结果按排名融合
   - 既保证语义相关性，又保证精确匹配的文档排在前面

3. **优雅降级**：
   - 如果 Dense 失败（API 不可用），可以只用 Sparse
   - 如果 Sparse 失败（索引缺失），可以只用 Dense

## 第二部分：对象和组件分析

### 2.1 DenseRetriever 核心对象

#### Class: `DenseRetriever`

**位置**：`src/core/query_engine/dense_retriever.py:23-232`

**职责**：
- 封装语义检索的完整流程
- 协调 Embedding Client 和 Vector Store
- 提供统一的检索接口

**关键属性**：

```python
class DenseRetriever:
    embedding_client: BaseEmbedding      # Embedding 提供者
    vector_store: BaseVectorStore        # 向量库
    default_top_k: int                   # 默认返回结果数
```

**输入**：
- `query: str` - 用户查询字符串
- `top_k: Optional[int]` - 返回结果数量
- `filters: Optional[Dict[str, Any]]` - 元数据过滤条件
- `trace: Optional[TraceContext]` - 追踪上下文（用于调试）

**输出**：
- `List[RetrievalResult]` - 检索结果列表，按相似度降序排列

**在检索流程中的位置**：
```
QueryProcessor → DenseRetriever → Vector Store → RetrievalResult[]
```

#### Function: `retrieve()`

**代码位置**：`dense_retriever.py:100-166`

**执行流程**：

```python
def retrieve(self, query, top_k, filters, trace):
    # Step 1: 验证输入
    self._validate_query(query)
    self._validate_dependencies()
    
    # Step 2: Embed 查询
    query_vectors = self.embedding_client.embed([query], trace=trace)
    query_vector = query_vectors[0]  # 取第一个（也是唯一一个）向量
    
    # Step 3: 向量库查询
    raw_results = self.vector_store.query(
        vector=query_vector,
        top_k=effective_top_k,
        filters=filters,
        trace=trace,
    )
    
    # Step 4: 转换为 RetrievalResult
    results = self._transform_results(raw_results)
    return results
```

**为什么这样设计**：
- **分离关注点**：Embedding 和 Vector Search 解耦，便于测试和替换
- **错误处理**：每个步骤都有独立的异常处理，错误信息清晰
- **可观测性**：通过 `trace` 参数记录每个阶段的耗时和结果

#### Configuration: `default_top_k`

**来源**：
1. 构造函数参数 `default_top_k`（默认 10）
2. `settings.retrieval.dense_top_k`（配置文件，默认 20）

**作用**：
- 当 `retrieve()` 调用时未指定 `top_k`，使用此默认值
- 控制检索的召回范围：`top_k` 越大，召回越多，但可能包含不相关文档

#### Embedding Model: `BaseEmbedding`

**抽象接口**：`src/libs/embedding/base_embedding.py`

**实现类**（根据配置选择）：
- `OpenAIEmbedding` - OpenAI text-embedding-3-small (1536 维)
- `AzureEmbedding` - Azure text-embedding-ada-002 (1536 维)
- `OllamaEmbedding` - Ollama nomic-embed-text (768 维)

**关键方法**：
```python
def embed(texts: List[str], trace=None) -> List[List[float]]:
    """批量生成 embedding"""
    # 返回: [[0.1, 0.2, ...], [0.3, 0.4, ...], ...]
```

**输入**：
- `texts: List[str]` - 文本列表（支持批量处理）

**输出**：
- `List[List[float]]` - 每个文本对应一个向量

**维度**：
- OpenAI/Azure: **1536 维**
- Ollama nomic-embed-text: **768 维**

**在检索流程中的位置**：
```
Query String → BaseEmbedding.embed() → Query Vector (1536-dim)
```

#### Vector Store: `BaseVectorStore`

**抽象接口**：`src/libs/vector_store/base_vector_store.py`

**实现类**：
- `ChromaStore` - ChromaDB（本地持久化向量库）

**关键方法**：
```python
def query(
    vector: List[float],
    top_k: int,
    filters: Optional[Dict[str, Any]] = None,
    trace: Optional[Any] = None,
) -> List[Dict[str, Any]]:
    """向量相似度搜索"""
    # 返回: [{"id": "chunk_id", "score": 0.85, "text": "...", "metadata": {...}}, ...]
```

**输入**：
- `vector: List[float]` - 查询向量（1536 维）
- `top_k: int` - 返回 Top-K 结果
- `filters: Dict` - 元数据过滤（如 `{"collection": "default"}`）

**输出**：
- `List[Dict]` - 原始结果，包含 `id`, `score`, `text`, `metadata`

**相似度计算**：
- ChromaDB 默认使用 **Cosine Similarity（余弦相似度）**
- 公式：`cos(θ) = (A·B) / (||A|| * ||B||)`
- 范围：`[-1, 1]`，值越大越相似

**在检索流程中的位置**：
```
Query Vector → ChromaStore.query() → Top-K Similar Chunks
```

#### Tokenizer: 无显式 Tokenizer

**为什么没有 Tokenizer？**

- Dense Retrieval **不需要分词**，因为：
  - Embedding 模型内部已经处理了文本（BERT/GPT 架构自带 tokenizer）
  - 我们只需要调用 `embedding_client.embed([query])`，模型内部会处理

**对比 Sparse Retrieval**：
- Sparse 需要显式分词（在 `QueryProcessor` 中完成）
- 因为 BM25 需要统计词频，必须知道"词"的边界

#### Query Encoder: `embedding_client.embed()`

**本质**：Embedding Model 就是 Query Encoder

**编码过程**（简化）：
```
"如何配置 Azure OpenAI？"
    ↓
[Tokenization] (模型内部)
    ↓
[Transformer Layers] (BERT/GPT 架构)
    ↓
[Pooling] (CLS token 或 mean pooling)
    ↓
[1536-dim Vector]
```

**关键点**：
- **批量处理**：虽然查询只有一条，但接口支持批量，便于未来优化
- **维度固定**：同一模型产生的向量维度固定（如 1536）

### 2.2 SparseRetriever 核心对象

#### Class: `SparseRetriever`

**位置**：`src/core/query_engine/sparse_retriever.py:23-289`

**职责**：
- 基于 BM25 倒排索引的关键词检索
- 从 Vector Store 获取完整文本和元数据
- 返回与 DenseRetriever 相同格式的 `RetrievalResult`

**关键属性**：

```python
class SparseRetriever:
    bm25_indexer: BM25Indexer           # BM25 索引器
    vector_store: BaseVectorStore        # 向量库（用于获取文本）
    default_top_k: int                   # 默认返回结果数
    default_collection: str              # 默认索引集合
```

**输入**：
- `keywords: List[str]` - 关键词列表（来自 QueryProcessor）
- `top_k: Optional[int]` - 返回结果数量
- `collection: Optional[str]` - BM25 索引集合名
- `trace: Optional[TraceContext]` - 追踪上下文

**输出**：
- `List[RetrievalResult]` - 检索结果列表，按 BM25 分数降序排列

**在检索流程中的位置**：
```
QueryProcessor → SparseRetriever → BM25Indexer → Vector Store → RetrievalResult[]
```

#### Function: `retrieve()`

**代码位置**：`sparse_retriever.py:103-185`

**执行流程**：

```python
def retrieve(self, keywords, top_k, collection, trace):
    # Step 1: 验证输入
    self._validate_keywords(keywords)
    self._validate_dependencies()
    
    # Step 2: 确保索引已加载
    if not self._ensure_index_loaded(effective_collection):
        return []  # 索引不存在，返回空
    
    # Step 3: 查询 BM25 索引
    bm25_results = self.bm25_indexer.query(
        query_terms=keywords,
        top_k=effective_top_k,
        trace=trace,
    )
    # bm25_results: [{"chunk_id": "...", "score": 2.5}, ...]
    
    # Step 4: 从向量库获取完整文本和元数据
    chunk_ids = [r["chunk_id"] for r in bm25_results]
    records = self.vector_store.get_by_ids(chunk_ids, trace=trace)
    # records: [{"id": "...", "text": "...", "metadata": {...}}, ...]
    
    # Step 5: 合并 BM25 分数和文本/元数据
    results = self._merge_results(bm25_results, records)
    return results
```

**为什么需要两步（BM25 + Vector Store）？**

1. **BM25 索引只存储分数和 chunk_id**，不存储完整文本
2. **Vector Store 存储完整文本和元数据**，但 BM25 已经帮我们筛选出了相关文档
3. **分离存储**：索引轻量（JSON），文本存储在向量库（ChromaDB）

#### BM25Indexer: `BM25Indexer`

**位置**：`src/ingestion/storage/bm25_indexer.py`

**职责**：
- 构建倒排索引（Inverted Index）
- 计算 IDF（Inverse Document Frequency）
- 执行 BM25 查询

**索引结构**（JSON 文件）：

```json
{
  "metadata": {
    "num_docs": 432,
    "avg_doc_length": 33.18,
    "total_terms": 5222,
    "collection": "default"
  },
  "index": {
    "machine": {
      "idf": 5.66,
      "df": 1,
      "postings": [
        {"chunk_id": "doc1_chunk0", "tf": 2, "doc_length": 76},
        {"chunk_id": "doc2_chunk1", "tf": 1, "doc_length": 45}
      ]
    },
    "learning": { ... }
  }
}
```

**BM25 公式**：

```
BM25(q, d) = Σ IDF(q_i) * (f(q_i, d) * (k1 + 1)) / (f(q_i, d) + k1 * (1 - b + b * |d| / avgdl))

其中：
- q = 查询词
- d = 文档
- f(q_i, d) = 词 q_i 在文档 d 中的词频（TF）
- |d| = 文档长度
- avgdl = 平均文档长度
- k1 = 1.5 (词频饱和度参数)
- b = 0.75 (长度归一化参数)
- IDF(q_i) = log((N - df + 0.5) / (df + 0.5))
  - N = 总文档数
  - df = 包含词 q_i 的文档数
```

**关键方法**：

```python

    # 4. 持久化到 JSON 文件

    """查询索引，返回 Top-K chunk_id 和分数"""

    # 2. 计算每个文档的 BM25 分数

```

**输入**：
- `query_terms: List[str]` - 查询关键词列表

**输出**：
- `List[Dict]` - `[{"chunk_id": "...", "score": 2.5}, ...]`

**在检索流程中的位置**：
```
Keywords → BM25Indexer.query() → Top-K Chunk IDs + Scores
```

```
│     ...                                                      │
│   ]                                                          │
└─────────────────────────────────────────────────────────────┘
```

#### 详细步骤解析

**Step 1: Query Input**

```python
query = "如何配置 Azure OpenAI？"
```

- **输入类型**：`str`
- **预处理**：`_validate_query(query)` 检查非空

**Step 2: Query Embedding**

```python
query_vectors = self.embedding_client.embed([query], trace=trace)
query_vector = query_vectors[0]  # 取第一个向量
```

**Embedding 过程**（以 OpenAI text-embedding-3-small 为例）：

1. **Tokenization**（模型内部）：
   ```
   "如何配置 Azure OpenAI？"
   → ["如何", "配置", "Azure", "OpenAI", "？"]
   → Token IDs: [1234, 5678, 9012, 3456, 7890]
   ```

2. **Transformer Encoding**：
   - 通过多层 Transformer 编码
   - 每个 token 得到一个向量表示

3. **Pooling**：
   - **Mean Pooling**：对所有 token 向量求平均
   - 或使用 **CLS token** 的向量

4. **输出**：
   ```
   [0.123, -0.456, 0.789, ..., 0.234]  # 1536 维向量
   ```

**Embedding 维度**：
- OpenAI/Azure: **1536 维**
- Ollama nomic-embed-text: **768 维**

**使用的 Embedding 模型**：
- 配置在 `config/settings.yaml`：
  ```yaml
  embedding:
    provider: "ollama"  # 或 "openai", "azure"
    model: "nomic-embed-text"  # 或 "text-embedding-3-small"
    dimensions: 768  # 或 1536
  ```

**Step 3: Vector Similarity Search**

```python
raw_results = self.vector_store.query(
    vector=query_vector,
    top_k=20,
    filters={"collection": "default"},
    trace=trace,
)
```

**ChromaDB 内部流程**：

1. **加载索引**：
   - 从 `data/db/chroma/` 加载向量索引
   - 索引结构：`{chunk_id: vector, metadata}`

2. **相似度计算**（Cosine Similarity）：
   ```
   similarity(chunk_i) = cosine(query_vector, chunk_i_vector)
                        = (query_vector · chunk_i_vector) / (||query_vector|| * ||chunk_i_vector||)
   ```

3. **排序**：
   - 按相似度降序排列
   - 相似度范围：`[-1, 1]`，通常 `[0, 1]`（归一化后）

4. **过滤**：
   - 应用 `filters`（如 `{"collection": "default"}`）
   - 只返回匹配的文档

5. **Top-K 选择**：
   - 取前 `top_k` 个结果

**为什么使用 Cosine Similarity？**

- **归一化**：不受向量长度影响，只关注方向
- **语义相似度**：两个语义相似的文本，即使长度不同，向量方向也相近
- **计算效率**：ChromaDB 使用 HNSW 索引加速

**Step 4: Result Transformation**

```python
def _transform_results(raw_results):
    results = []
    for raw in raw_results:
        result = RetrievalResult(
            chunk_id=str(raw.get('id', '')),
            score=float(raw.get('score', 0.0)),
            text=str(raw.get('text', '')),
            metadata=raw.get('metadata', {}),
        )
        results.append(result)
    return results
```

**为什么需要转换？**

- **统一接口**：Vector Store 返回的是 `Dict`，需要转换为 `RetrievalResult` 对象
- **类型安全**：`RetrievalResult` 有类型验证（`__post_init__`）
- **下游兼容**：HybridSearch、Reranker 都期望 `RetrievalResult` 对象

**BM25 计算过程**：

1. **查找倒排索引**：
   ```
   "如何" → postings: [chunk_001, chunk_005]
   "配置" → postings: [chunk_001, chunk_003]
   "Azure" → postings: [chunk_001, chunk_002, chunk_003]
   "OpenAI" → postings: [chunk_001, chunk_002]
   ```

2. **计算每个文档的 BM25 分数**：
   ```
   BM25(chunk_001) = 
     IDF("如何") * TF("如何", chunk_001) * (k1+1) / (TF + k1*(1-b+b*|d|/avgdl)) +
     IDF("配置") * TF("配置", chunk_001) * (k1+1) / (TF + k1*(1-b+b*|d|/avgdl)) +
     IDF("Azure") * TF("Azure", chunk_001) * (k1+1) / (TF + k1*(1-b+b*|d|/avgdl)) +
     IDF("OpenAI") * TF("OpenAI", chunk_001) * (k1+1) / (TF + k1*(1-b+b*|d|/avgdl))
   ```

3. **排序并返回 Top-K**：
   ```
   [{"chunk_id": "chunk_001", "score": 2.45},
    {"chunk_id": "chunk_002", "score": 1.89},
    ...]
   ```

**BM25 参数**：
- `k1 = 1.5`：词频饱和度参数（控制 TF 的影响）
- `b = 0.75`：长度归一化参数（控制文档长度的影响）

```python

    self,
    raw_results: List[Dict[str, Any]],

    results = []
    for raw in raw_results:
        try:
            result = RetrievalResult(
                chunk_id=str(raw.get('id', '')),
                score=float(raw.get('score', 0.0)),
```

- **部分失败不影响整体**：如果某个结果格式错误，不影响其他结果
- **可观测性**：记录 warning，便于调试

```python
# BM25Indexer.query() 内部
scores = {}  # {chunk_id: score}
for term in query_terms:
    if term in index:
        idf = index[term]["idf"]
        for posting in index[term]["postings"]:
            chunk_id = posting["chunk_id"]
            tf = posting["tf"]
            doc_length = posting["doc_length"]
            # 计算 BM25 分数
            score = idf * (tf * (k1 + 1)) / (tf + k1 * (1 - b + b * doc_length / avgdl))
            scores[chunk_id] = scores.get(chunk_id, 0) + score

# 按分数排序，返回 Top-K
sorted_results = sorted(scores.items(), key=lambda x: x[1], reverse=True)[:top_k]
return [{"chunk_id": cid, "score": score} for cid, score in sorted_results]
```

## 第五部分：关键参数解析

### 5.1 Dense Retrieval 参数

#### `top_k` (默认: 20)

**作用**：
- 控制返回的检索结果数量
- 影响召回率和精确率的平衡

**影响**：
- **`top_k` 太小**（如 5）：
  - ✅ 精确率高（只返回最相关的）
  - ❌ 召回率低（可能漏掉相关文档）
- **`top_k` 太大**（如 100）：
  - ✅ 召回率高（包含更多相关文档）
  - ❌ 精确率低（包含不相关文档）
  - ❌ 计算成本高（需要处理更多结果）

**调优建议**：
- **初始值**：20（配置文件 `settings.yaml`）
- **根据场景调整**：
  - **精确查询**（如代码搜索）：`top_k = 5-10`
  - **探索性查询**（如知识问答）：`top_k = 20-50`
  - **Reranker 前**：`top_k = 50-100`（让 Reranker 筛选）

**代码位置**：
```python
# config/settings.yaml
retrieval:
  dense_top_k: 20
```

#### `similarity_metric` (默认: Cosine)

**当前实现**：ChromaDB 默认使用 **Cosine Similarity**

**公式**：
```
cos(θ) = (A·B) / (||A|| * ||B||)
```

**范围**：`[-1, 1]`，通常归一化到 `[0, 1]`

**为什么选择 Cosine？**

- **归一化**：不受向量长度影响，只关注方向
- **语义相似度**：两个语义相似的文本，向量方向相近
- **计算效率**：ChromaDB 使用 HNSW 索引加速

**其他选项**（ChromaDB 支持）：
- **Dot Product**：`A·B`，适合归一化向量
- **Euclidean (L2)**：`||A - B||`，适合非归一化向量

**如何修改**：
```python
# ChromaStore 初始化时
collection = client.get_or_create_collection(
    name="knowledge_hub",
    metadata={"hnsw:space": "cosine"}  # 或 "l2", "ip"
)
```

#### `embedding_model` (配置: Ollama nomic-embed-text)

**当前配置**：
```yaml
embedding:
  provider: "ollama"
  model: "nomic-embed-text"
  dimensions: 768
```

**模型对比**：

| 模型 | 维度 | 优势 | 劣势 |
|------|------|------|------|
| **nomic-embed-text** | 768 | 本地运行、免费、低延迟 | 语义理解可能不如 OpenAI |
| **text-embedding-3-small** | 1536 | 语义理解强、多语言支持好 | 需要 API、有成本 |
| **text-embedding-ada-002** | 1536 | 稳定、广泛使用 | 较旧、性能不如 3-small |

**如何选择**：
- **本地部署**：Ollama（免费、低延迟）
- **生产环境**：OpenAI/Azure（语义理解更强）
- **多语言**：OpenAI（支持更多语言）

**维度影响**：
- **维度越高**：表达能力越强，但存储和计算成本越高
- **768 vs 1536**：1536 通常语义理解更好，但 768 已经足够大多数场景

#### `chunk_size` 和 `chunk_overlap` (Ingestion 阶段)

**配置位置**：
```yaml
ingestion:
  chunk_size: 1000
  chunk_overlap: 200
```

**作用**：
- **`chunk_size`**：每个 chunk 的最大字符数
- **`chunk_overlap`**：相邻 chunk 之间的重叠字符数

**影响检索质量**：
- **`chunk_size` 太小**（如 200）：
  - ❌ 上下文不完整
  - ❌ 可能切分重要信息
- **`chunk_size` 太大**（如 5000）：
  - ❌ 包含无关信息
  - ❌ Embedding 可能被稀释
- **`chunk_overlap` 太小**（如 0）：
  - ❌ 可能切分句子/段落
- **`chunk_overlap` 太大**（如 500）：
  - ❌ 冗余存储和计算

**调优建议**：
- **通用文档**：`chunk_size=1000, overlap=200`
- **技术文档**：`chunk_size=1500, overlap=300`（代码块可能较长）
- **对话记录**：`chunk_size=500, overlap=100`（对话较短）

#### `index_type` (ChromaDB)

**当前实现**：ChromaDB（本地持久化）

**索引算法**：HNSW (Hierarchical Navigable Small World)

**优势**：
- **快速查询**：O(log N) 时间复杂度
- **本地存储**：无需外部服务
- **增量更新**：支持动态添加文档

**其他选项**：
- **FAISS**：Facebook 的向量库，性能更强但需要手动管理
- **Pinecone**：托管服务，无需运维但需要付费
- **Qdrant**：开源向量库，支持更多高级功能

**作用**：与 Dense 相同，控制返回结果数

**调优建议**：
- **初始值**：20
- **与 Dense 配合**：通常 `sparse_top_k = dense_top_k`，便于 RRF Fusion

#### `k1` 和 `b` (BM25 参数)

**默认值**：
```python
k1 = 1.5  # 词频饱和度参数
b = 0.75  # 长度归一化参数
```

**`k1` 的作用**：
- **控制词频（TF）的影响**
- **`k1` 越大**：词频影响越大（高频词得分更高）
- **`k1` 越小**：词频影响越小（更关注 IDF）

**调优建议**：
- **通用场景**：`k1 = 1.5`（默认值）
- **短文档**（如标题、摘要）：`k1 = 2.0`（更重视词频）
- **长文档**（如论文、报告）：`k1 = 1.2`（更重视 IDF）

**`b` 的作用**：
- **控制文档长度归一化**
- **`b = 0`**：不进行长度归一化
- **`b = 1`**：完全归一化（长文档和短文档得分相同）

**调优建议**：
- **通用场景**：`b = 0.75`（默认值）
- **短文档为主**：`b = 0.5`（减少长度惩罚）
- **长文档为主**：`b = 1.0`（完全归一化）

**代码位置**：
```python

self.b = 0.75
```

### 6.1 Embedding 优化

#### 当前实现

```python
query_vectors = self.embedding_client.embed([query], trace=trace)
```

**问题**：
- **单条查询**：每次只处理一条查询，无法利用批量处理的优势

#### 优化方案 1：批量查询缓存

**场景**：如果多个用户查询相同或相似的问题

**实现**：
```python
from functools import lru_cache

@lru_cache(maxsize=1000)
def _cached_embed(self, query: str) -> List[float]:
    vectors = self.embedding_client.embed([query])
    return vectors[0]
```

**效果**：
- **减少 API 调用**：相同查询直接返回缓存
- **降低延迟**：缓存命中时 < 1ms

**权衡**：
- **内存占用**：1000 个查询 × 1536 维 × 4 字节 ≈ 6MB
- **缓存失效**：如果 Embedding 模型更新，需要清空缓存

#### 优化方案 2：异步 Embedding

**场景**：如果 Embedding API 延迟较高（如 OpenAI API）

**实现**：

```python
import asyncio

async def retrieve_async(self, query: str, ...):

```

**效果**：
- **并发处理**：可以同时处理多个查询
- **降低延迟**：不阻塞其他请求

**权衡**：
- **复杂度**：需要异步框架支持
- **资源消耗**：并发请求可能触发 API 限流
